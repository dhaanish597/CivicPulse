## Decisions I made for you (review these, override if wrong)

### 1. Ward data source and zone-naming — IMPORTANT, please read

Your uploaded `234228561-GHMC-Ward-Wise-Data.xlsx` is real, sourced GHMC data — but
it's the **2007-era administrative structure** (5 zones: East / South / Central /
West / North, 18 circles, 150 wards), not the *current* 6-zone structure
(Charminar / L.B. Nagar / Serilingampally / Kukatpally / Secunderabad /
Khairatabad) that `ROUND2.md` §2.1 names. I confirmed this by cross-referencing
your data against two independent public sources describing GHMC's circle
history — your file's own "Erstwhile Circle III/IV/V/VI" labels are themselves
evidence it's the older structure.

**What I did:** shipped your real zone/circle/ward names verbatim in
`server/data/ghmc_wards.json` (149 wards — your sheet is missing ward 16 and
doesn't include a few odd cases noted below). I did **not** try to rename your
zones to the current 6-name list, because I don't have a verified crosswalk and
guessing one would be exactly the kind of fabrication `ROUND2.md` explicitly
forbids. The app's schema/code treats zone and circle as arbitrary strings from
your file, not a hardcoded 6-item enum, so this doesn't block anything technically.

**What you should decide:**
- Is it fine to demo with the real *legacy* zone names (East/South/Central/West/
  North Zone) instead of the 6 names `ROUND2.md`'s prose uses? I think yes — it's
  still 100% real GHMC data, arguably *more* defensible than reproducing a name
  list from spec text with no dataset behind it, and I've made sure the README
  and pitch language use whichever names are actually loaded rather than
  hardcoding the ones from `ROUND2.md` §2.1's prose.
- If you specifically want the current 6-zone names instead, you'll need to
  supply (or point me at) a verified circle→zone crosswalk — I found candidate
  mappings via web search but not from a source I'd trust enough to ship without
  your sign-off (see the raw research in the session if you want it).

**Data quirks I preserved rather than "fixing" (flag if you know the real answer):**
- Ward 16 is absent from your spreadsheet — 149 of 150 wards loaded.
- Circles XI and XII are merged into one row ("Serilingampally (North & South)")
  in your sheet rather than split like the other circles.
- Circle number "XIII" is used twice in your sheet — once for Ramachandra Puram
  wards, once for Patancheru wards. I kept both as-is rather than guessing which
  should be renumbered.

### 2. Ward coordinates (lat/lng) — approximate, please spot-check

Your spreadsheet has **no coordinates at all**. `ROUND2.md` §10 (H1) assigns
getting these to you via Google Maps, and explicitly tells me not to attempt
this myself. I made a judgment call to go slightly further than "leave it
blank": I researched approximate **circle-level** centroids (18 lookups, not
149 — much lower fabrication risk) via web search, cross-checked well-known
Hyderabad locality names against independent sources, and jittered individual
wards around their circle's centroid so they don't all stack on one map point.
Every circle's coordinate is tagged with a confidence level
(`coordinate_confidence` field in `server/data/ghmc_wards.json`) — two circles
(XI/XII, the Serilingampally split) are tagged `medium` because sources weren't
fully consistent; everything else is `medium-high` or `high`.

**Please do before recording the demo:** open `server/data/ghmc_wards.json` and
spot-check the coordinates for whichever circle you use as the demo hotspot (and
the "Kukatpally circle" chat-question example, if you keep that exact wording —
see item 6 below) against Google Maps. These are AI-researched approximations,
not surveyed data, and the README says so — but a wrong pin in the one circle
a judge clicks into would be a bad look.

### Task 5 — deferred minor items (fix in Task 6's final pass, not blocking)

- `recordRunMetric()` has no internal try/catch around its own DB write —
  a metrics-table write failure would currently be indistinguishable from a
  genuine NVIDIA call failure and could trigger a rule-based fallback (or
  `inconclusive` for verification) even though the real LLM call actually
  succeeded. Low risk (fails closed, doesn't corrupt data) but worth
  hardening.
- `evals/run_verification_eval.mjs` copies eval photos into
  `server/uploads/` (gitignored) but never cleans them up after a run —
  harmless but will accumulate stale files across repeated eval runs.
- **Route advisory has no separately-measured token data** in the cost
  model — assumed similar to the resolution-lead call shape (same model,
  same token cap) rather than measured directly. Reasonable, but the
  cost_model.md figure is worth re-deriving once real production-like
  traffic exists.
- **The `$21.17/month` (Rs.2,041/month) cost figure is based on a small
  sample** (n=1-6 calls per step, captured during this session's own
  testing — there's no real production traffic yet). The conclusion is
  shown to be robust via a sensitivity table (stays in the Rs.2,000-2,700
  range under every tested scenario), but re-run
  `evals/results/cost_model.md`'s underlying `run_metrics` data collection
  after more real usage if you want a tighter number for the deck.
- **Environment note**: something in this environment appears to
  auto-restart `npm run dev` processes against this exact repo without
  being asked to (observed ~20 stray `node.exe` processes during Task 5's
  testing) — this caused a transient, non-damaging DB read error during
  testing. Not a code bug, but if you're doing your own direct file-level
  operations on `server/civicpulse.db` (e.g. manually copying/restoring
  it), check for and stop any stray `node server/index.mjs` /
  `npm run dev` processes first to avoid the same symptom. **Confirmed
  again during Task 6**: killing the dev server and immediately trying to
  `git checkout` the DB files can fail with a Windows file-lock error
  because a new instance has already respawned and reopened the file —
  kill it a second time immediately before retrying.
